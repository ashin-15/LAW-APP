## 2024-05-18 - Missing Field Validation in Firestore Allow Create Leads to Privilege Escalation
**Vulnerability:** The `firestore.rules` file allowed any authenticated user to create a document in the `laws` collection, but didn't restrict what data could be inserted. An attacker could bypass the admin review process by manually inserting a document with `status: 'approved'` and `approvedBy: 'hacked'`, immediately publishing a law to the global approved laws list without review.
**Learning:** Firestore's `allow create` rule allows writing ANY data by default if the condition evaluates to true. Missing validation on document payloads during creation allows privilege escalation when business logic relies on trustable metadata fields (like `status`).
**Prevention:** Always use `request.resource.data` assertions in Firestore rules to restrict which states a newly created document can have, particularly for fields governing approval flows or visibility controls. Example: `request.resource.data.status == 'pending'`.
